03 Phase 2 — Batched Migration Loading
◈ Contributed — Migration Execution

10M+ rows cannot be migrated in a single transaction — log growth, lock pressure, and restart risk make that impractical. I helped implement a batched idempotent load pattern that moves controlled chunks with progress tracking and safe re-run capability.

T-SQL — Batch Migration PatternSimulated pattern — no proprietary code
DECLARE @BatchSize int = 100000;

WHILE 1=1
BEGIN
  INSERT INTO dbo.Target (BusinessKey, Col1, Col2, Col3)
  SELECT TOP (@BatchSize) s.BusinessKey, s.Col1, s.Col2, s.Col3
  FROM dbo.Source s
  WHERE NOT EXISTS (
    SELECT 1 FROM dbo.Target t WHERE t.BusinessKey = s.BusinessKey
  )
  ORDER BY s.BusinessKey;

  IF @@ROWCOUNT = 0 BREAK;
END
04 Phase 3 — Reconciliation & Validation
✦ I Directly Worked On This

Validation was multi-layered: row counts, sum checks on financial fields, and spot sampling of high-value records. All three layers together make "zero material data loss" defensible — not just a claim.

T-SQL — Three-Layer Validation PatternSimulated pattern
-- Layer 1: Row count reconciliation
SELECT 'Source' AS Side, COUNT(*) AS Cnt FROM dbo.Source
UNION ALL
SELECT 'Target', COUNT(*) FROM dbo.Target;

-- Layer 2: Financial sum reconciliation
SELECT SUM(Outstanding_Balance) AS SrcSum FROM dbo.Source;
SELECT SUM(Outstanding_Balance) AS TgtSum FROM dbo.Target;

-- Layer 3: Spot-check high-value records
SELECT TOP 50 * FROM dbo.Target
ORDER BY Outstanding_Balance DESC;
05 Phase 4 — Post-Migration Index Strategy
✦ I Directly Led This

Bulk loads reset physical layout and statistics. Post-migration indexes must be rebuilt aligned to actual workload patterns — not speculative, not copied from the old system blindly. I profiled the top reporting queries and implemented targeted indexes.

T-SQL — Index Strategy & Statistics RefreshSimulated pattern
-- Refresh statistics after bulk load (critical for plan quality)
UPDATE STATISTICS dbo.Target WITH FULLSCAN;

-- Covering index for top reporting predicates
CREATE NONCLUSTERED INDEX IX_Target_Report
ON dbo.Target (AsOfDate, Region, LoanType)
INCLUDE (Outstanding_Balance, Risk_Score, Default_Flag);

-- Detect top regressions post-migration
SELECT TOP 25
  qs.total_elapsed_time / NULLIF(qs.execution_count,0) AS avg_elapsed,
  qs.total_logical_reads / NULLIF(qs.execution_count,0) AS avg_reads,
  SUBSTRING(st.text,(qs.statement_start_offset/2)+1,
    ((CASE qs.statement_end_offset WHEN -1 THEN DATALENGTH(st.text)
      ELSE qs.statement_end_offset END - qs.statement_start_offset)/2)+1) AS stmt
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) st
ORDER BY avg_elapsed DESC;
06 Phase 5 — Query Tuning for 30% Improvement
✦ I Directly Led This

Indexes alone rarely produce 30% gains. The complementary workstream was execution plan analysis and query refactoring — finding and fixing anti-patterns that prevent the optimizer from using the indexes effectively.

T-SQL — SARGable Predicate Pattern (Before vs After)Simulated pattern
-- BEFORE: function on column forces scan, ignores index
-- WHERE CONVERT(date, TransactionDate) = @d

-- AFTER: range predicate enables index seek
WHERE TransactionDate >= @d
  AND TransactionDate < DATEADD(day, 1, @d)

-- Window function replacing correlated subquery (5x faster on 10M rows)
-- BEFORE: correlated subquery ran in ~8 seconds
-- AFTER: window function with partition
SELECT
  LoanID,
  RiskScore,
  AVG(RiskScore) OVER (PARTITION BY LoanType) AS AvgByType,
  RiskScore - AVG(RiskScore) OVER (PARTITION BY LoanType) AS Deviation
FROM dbo.Target
WHERE AsOfDate = @AsOfDate;
Why 30% is Defensible

Bulk-load migrations commonly degrade performance because statistics are stale and indexes are absent for new workload patterns. A targeted post-migration tuning sprint — profiling, indexing, plan analysis, and SARGability fixes — regularly produces 20-40% improvements on previously unoptimized workloads.
