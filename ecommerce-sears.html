02 Phase 1 — Performance Baseline & Bottleneck Identification
✦ I Directly Led This

Before tuning anything, I established where the time was actually going. For an e-commerce system, the three failure modes that cause visible downtime symptoms are: slow queries degrading under peak traffic, SQL Agent job failures silently breaking ETL/reporting, and disk space pressure from unchecked growth. I built a diagnostic process covering all three.

T-SQL — Bottleneck Diagnostic QueriesSimulated pattern — no proprietary code
-- Long-running requests (real-time triage)
SELECT TOP 25
  r.session_id, r.status, r.total_elapsed_time,
  r.cpu_time, r.logical_reads,
  SUBSTRING(t.text,(r.statement_start_offset/2)+1,
    ((CASE r.statement_end_offset WHEN -1 THEN DATALENGTH(t.text)
      ELSE r.statement_end_offset END - r.statement_start_offset)/2)+1) AS stmt
FROM sys.dm_exec_requests r
CROSS APPLY sys.dm_exec_sql_text(r.sql_handle) t
WHERE r.session_id != @@SPID
ORDER BY r.total_elapsed_time DESC;

-- Top system waits (distinguish CPU vs IO vs lock pressure)
SELECT TOP 20 wait_type, wait_time_ms,
  wait_time_ms / NULLIF(waiting_tasks_count,0) AS avg_wait_ms
FROM sys.dm_os_wait_stats
WHERE wait_type NOT LIKE '%SLEEP%'
ORDER BY wait_time_ms DESC;
03 Phase 2 — Query & Index Optimization (40% Downtime Reduction)
✦ I Directly Led This

Most downtime in SQL Server environments is caused by slow queries that cascade into blocking chains. Fixing the top offenders — SARGability issues, missing covering indexes, key lookup elimination — produced measurable reduction in both latency and incident frequency.

T-SQL — Index Strategy & SARGable PatternsSimulated pattern
-- Covering index for order history queries
CREATE NONCLUSTERED INDEX IX_Orders_Customer_Date
ON dbo.Orders (CustomerID, OrderDate)
INCLUDE (OrderID, OrderStatus, OrderTotal);

-- Eliminate key lookup on order items
CREATE NONCLUSTERED INDEX IX_OrderItems_OrderID
ON dbo.OrderItems (OrderID)
INCLUDE (SKU, Quantity, UnitPrice);

-- SARGable date filter (seek instead of scan)
-- BAD:  WHERE CONVERT(date, OrderDate) = @d
-- GOOD:
WHERE OrderDate >= @d
  AND OrderDate < DATEADD(day, 1, @d)
04 Phase 3 — Operational Hardening (99.9% Uptime)
✦ I Directly Implemented This

High uptime over 10 years is not accidental. It requires systematic operational hygiene: backup verification (not just scheduling), integrity checks, space monitoring, and job health monitoring so failures are caught before they cascade.

T-SQL — Operational Health MonitoringSimulated pattern
-- Verify backup is readable (run after every backup job)
RESTORE VERIFYONLY FROM DISK = 'X:\Backups\ECommerceDB.bak';

-- Refresh stats and check integrity (scheduled off-peak)
UPDATE STATISTICS dbo.Orders WITH FULLSCAN;
DBCC CHECKDB('ECommerceDB') WITH NO_INFOMSGS;

-- Top tables by size (capacity planning)
SELECT TOP 20
  t.name AS TableName,
  SUM(a.total_pages) * 8 / 1024.0 AS SizeMB
FROM sys.tables t
JOIN sys.indexes i     ON t.object_id=i.object_id
JOIN sys.partitions p  ON i.object_id=p.object_id AND i.index_id=p.index_id
JOIN sys.allocation_units a ON p.partition_id=a.container_id
WHERE t.is_ms_shipped=0
GROUP BY t.name
ORDER BY SizeMB DESC;

-- Last failed SQL Agent jobs
SELECT TOP 20 j.name, h.run_date, h.run_status, h.message
FROM msdb.dbo.sysjobhistory h
JOIN msdb.dbo.sysjobs j ON h.job_id=j.job_id
WHERE h.step_id=0 AND h.run_status != 1
ORDER BY h.run_date DESC;
05 Phase 4 — Reporting Accuracy (+15%)
✦ I Directly Worked On This

Reporting accuracy problems in e-commerce usually trace back to ETL issues — orphaned records, datatype drift, or incomplete loads creating double-counts. I implemented reconciliation checks and ETL quality controls that closed these gaps.

T-SQL — Reporting Reconciliation & Data QualitySimulated pattern
-- Reconcile daily revenue between OLTP and reporting layer
SELECT CAST(OrderDate AS date) AS [Date],
       SUM(OrderTotal) AS OLTP_Revenue
FROM dbo.Orders WHERE OrderStatus='Completed'
GROUP BY CAST(OrderDate AS date);

SELECT [Date], SUM(Revenue) AS DW_Revenue
FROM dbo.DailySalesSummary GROUP BY [Date];

-- Detect orphaned order items (accuracy killer)
SELECT oi.OrderID, COUNT(*) AS OrphanedItems
FROM dbo.OrderItems oi
LEFT JOIN dbo.Orders o ON o.OrderID=oi.OrderID
WHERE o.OrderID IS NULL
GROUP BY oi.OrderID;
