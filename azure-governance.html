Running in parallel was a data governance initiative to improve data accuracy across departments, which had suffered from duplicate records and inconsistent ETL validation controls.

My contributions spanned the full lifecycle: SSMA-based compatibility assessment, schema changes for Azure SQL readiness, hands-on duplicate record remediation (150 records removed), ETL quality gate implementation, and post-migration monitoring setup. The data governance work contributed to a 50% improvement in data accuracy across departments.

02 Phase 1 — SMS Compatibility Assessment (SSMA)
◈ Contributed — Assessment Workstream

I used SQL Server Migration Assistant (SSMA) to run a compatibility assessment on the Student Management System database — cataloging unsupported features, T-SQL syntax differences, cross-database dependencies, and any objects that would need remediation before Azure SQL migration. Issues were triaged into Must-Fix, Should-Fix, and Can-Defer categories.

T-SQL — Pre-Migration Compatibility ChecksSimulated pattern — no proprietary code
-- Check for datatype drift that would need conversion
SELECT t.name AS TableName, c.name AS ColName, tp.name AS DataType
FROM sys.columns c
JOIN sys.tables t ON t.object_id = c.object_id
JOIN sys.types tp ON tp.user_type_id = c.user_type_id
WHERE tp.name IN ('text','ntext','image')  -- legacy types incompatible
ORDER BY t.name;

-- Detect truncation risk before VARCHAR conversion
SELECT TOP 50 *
FROM dbo.Student
WHERE LEN(Notes) > 4000; -- exceeds target column size
03 Phase 2 — Duplicate Student Record Cleanup
✦ I Directly Built & Executed This

SMS data had 150 duplicate student records — caused by data entry inconsistencies and system migration artifacts. I developed T-SQL scripts to detect them using a match-key approach, produced a review list for business sign-off, then executed a controlled archive-before-delete workflow.

T-SQL — Duplicate Detection & Controlled CleanupSimulated pattern
-- Step 1: Find duplicates using match key
WITH Candidates AS (
  SELECT StudentID,
    LOWER(LTRIM(RTRIM(FirstName))) AS fn,
    LOWER(LTRIM(RTRIM(LastName)))  AS ln,
    DOB, Email, CreatedDate,
    ROW_NUMBER() OVER (
      PARTITION BY LOWER(LTRIM(RTRIM(FirstName))),
                   LOWER(LTRIM(RTRIM(LastName))), DOB, Email
      ORDER BY CreatedDate DESC
    ) AS rn
  FROM dbo.Student
)
SELECT * FROM Candidates WHERE rn > 1;  -- review list

-- Step 2: Archive before delete (safety first)
BEGIN TRAN;
  INSERT INTO dbo.Student_DedupeArchive (StudentID, ArchivedAt)
  SELECT StudentID, SYSUTCDATETIME()
  FROM Candidates WHERE rn > 1;

  DELETE s FROM dbo.Student s
  JOIN Candidates c ON c.StudentID = s.StudentID AND c.rn > 1;
COMMIT TRAN;
Result

150 duplicate student records removed. Archive table preserved all removed records for 90-day review window before permanent disposal per data retention policy.

04 Phase 3 — ETL Quality Gates & Data Governance
✦ I Directly Built This

One-time cleanup fixes the past. ETL quality gates prevent recurrence. I embedded validation rules inside SSIS pipelines: completeness checks, domain validation, uniqueness checks, and referential integrity — with bad rows quarantined to a reject table with reason codes rather than silently loaded.

T-SQL — ETL Quality Gate Schema & Validation RulesSimulated pattern
-- Reject table: makes quality issues measurable
CREATE TABLE dbo.ETL_Rejects (
  RejectID    bigint IDENTITY(1,1) PRIMARY KEY,
  EntityName  nvarchar(50),
  NaturalKey  nvarchar(200),
  RejectReason nvarchar(200),
  LoadBatchID nvarchar(50),
  RejectedAt  datetime2 DEFAULT SYSUTCDATETIME()
);

-- Validation: student must have valid active email
INSERT INTO dbo.ETL_Rejects(EntityName,NaturalKey,RejectReason,LoadBatchID)
SELECT 'Student', CAST(StudentID AS nvarchar), 'Missing primary email', @BatchID
FROM stg.Student
WHERE PrimaryEmail IS NULL OR LEN(PrimaryEmail) < 5;

-- Validation: DOB must be realistic
INSERT INTO dbo.ETL_Rejects(EntityName,NaturalKey,RejectReason,LoadBatchID)
SELECT 'Student', CAST(StudentID AS nvarchar), 'Invalid DOB', @BatchID
FROM stg.Student
WHERE DOB > DATEADD(year, -10, GETDATE())
   OR DOB < DATEADD(year, -100, GETDATE());
05 Phase 4 — Azure SQL Monitoring Setup
✦ I Directly Implemented This

Post-migration, I configured monitoring baselines in Azure SQL — top query performance, DTU/vCore utilization, storage trends, and blocking indicators. These baselines feed the next optimization wave and provide proof that the migration didn't regress performance.

T-SQL — Azure SQL Performance Baseline QueriesSimulated pattern
-- Top CPU queries (Azure SQL DMV pattern)
SELECT TOP 25
  qs.total_worker_time / NULLIF(qs.execution_count,0) AS avg_cpu,
  qs.execution_count,
  qs.total_logical_reads / NULLIF(qs.execution_count,0) AS avg_reads,
  SUBSTRING(st.text,(qs.statement_start_offset/2)+1,
    ((CASE qs.statement_end_offset WHEN -1 THEN DATALENGTH(st.text)
      ELSE qs.statement_end_offset END - qs.statement_start_offset)/2)+1) AS stmt
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) st
ORDER BY avg_cpu DESC;

-- Storage growth trend (capacity planning)
SELECT name, type_desc, size * 8 / 1024.0 AS SizeMB
FROM sys.database_files
ORDER BY SizeMB DESC;
