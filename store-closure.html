TransformCo's restructuring involved closing 50+ Sears and Kmart retail locations over three years. Each store closure carried real data risk: POS transaction history, inventory snapshots, and customer records had to be migrated to central corporate systems, validated for completeness, and archived for future dispute resolution and compliance — before the local systems could be decommissioned.

I led the data execution side of this program: profiling and cleaning store-level data before migration, designing the archive schema for future accessibility, running idempotent batched migrations with three-layer validation, and maintaining closure progress reporting for leadership. The result was zero material data loss across all 50+ closures.

Design Context

Each store closure followed the same five-phase workflow. Decommission approval was gated behind validation sign-off — no store system was shut down until row counts, sum reconciliation, and spot sampling all confirmed data integrity.

02 Phase 1 — Closure Intake & Scope Control
✦ I Led Closure Execution Activities

Each store closure was treated as a mini-project with its own scope checklist. Before any data moved, I confirmed: what systems were in scope (POS, inventory, transactions), data retention requirements (how many years must stay queryable for disputes/audits), the final extraction window, and rollback criteria — "if validation fails, source stays online."

  • Systems in scope — POS transaction headers and line items, inventory snapshots, customer transaction history
  • Retention policy — confirmed years for active query access vs cold archive
  • Extraction window — coordinated with operations for end-of-day final pull
  • Rollback gate — validation must pass before source decommission is approved
03 Phase 2 — Data Profiling & Cleanup
✦ I Directly Executed This

Store-level databases accumulated integrity issues over years — duplicate transactions from replication glitches, inconsistent region codes, mismatched time zones on transaction timestamps. Cleaning before migrating means central systems don't inherit the garbage.

T-SQL — Pre-Migration Duplicate DetectionSimulated pattern — no proprietary code
-- Find duplicate transactions by natural key
SELECT StoreID, TransactionID, COUNT(*) AS c
FROM dbo.PosTransactionHeader
GROUP BY StoreID, TransactionID
HAVING COUNT(*) > 1;

-- Deduplicate using ROW_NUMBER (keep most recent)
WITH d AS (
  SELECT *,
    ROW_NUMBER() OVER (
      PARTITION BY StoreID, TransactionID
      ORDER BY LastUpdated DESC
    ) AS rn
  FROM dbo.PosTransactionHeader
)
DELETE FROM d WHERE rn > 1;
04 Phase 3 — Archive Schema Design
✦ I Directly Designed This

Archive tables were designed for future accessibility, not just long-term storage. Dispute resolution, tax audits, and regulatory inquiries can happen years after a store closes — the archive must support targeted queries by store, date, and transaction.

T-SQL — Archive Schema & Access IndexSimulated pattern
CREATE TABLE dbo.Archive_PosTransactions (
  ArchiveBatchID   bigint NOT NULL,
  StoreID          int NOT NULL,
  TransactionID    bigint NOT NULL,
  BusinessDate     date NOT NULL,
  TotalAmount      decimal(18,2) NOT NULL,
  CustomerID       bigint NULL,
  SourceSystem     nvarchar(50) NOT NULL,
  ArchivedAt       datetime2 NOT NULL DEFAULT SYSUTCDATETIME(),
  PRIMARY KEY (StoreID, TransactionID, ArchiveBatchID)
);

-- Index for future dispute/audit queries
CREATE INDEX IX_Archive_ByDate
ON dbo.Archive_PosTransactions (BusinessDate, StoreID)
INCLUDE (CustomerID, TotalAmount);
05 Phase 4 — Batched Extraction & Zero-Loss Validation
✦ I Directly Executed This

Extraction used idempotent batch loading — each batch checked whether records already existed before inserting, making restarts safe if a store system went offline mid-run. Validation followed a three-layer protocol: row counts, sum reconciliation on financial fields, and high-value spot sampling.

T-SQL — Idempotent Batch Load & Three-Layer ValidationSimulated pattern
-- Idempotent batch migration
DECLARE @Batch int = 200000;
WHILE 1=1
BEGIN
  INSERT INTO dbo.Archive_PosTransactions
    (ArchiveBatchID,StoreID,TransactionID,BusinessDate,TotalAmount,CustomerID,SourceSystem)
  SELECT TOP(@Batch)
    @BatchID,StoreID,TransactionID,BusinessDate,TotalAmount,CustomerID,'POS'
  FROM dbo.PosTransactionHeader s
  WHERE s.StoreID = @StoreID
    AND NOT EXISTS(SELECT 1 FROM dbo.Archive_PosTransactions a
                   WHERE a.StoreID=s.StoreID AND a.TransactionID=s.TransactionID)
  ORDER BY s.TransactionID;
  IF @@ROWCOUNT=0 BREAK;
END

-- Layer 1: Row count
SELECT COUNT(*) FROM dbo.PosTransactionHeader WHERE StoreID=@StoreID;
SELECT COUNT(*) FROM dbo.Archive_PosTransactions WHERE StoreID=@StoreID;

-- Layer 2: Financial sum reconciliation
SELECT SUM(TotalAmount) FROM dbo.PosTransactionHeader WHERE StoreID=@StoreID;
SELECT SUM(TotalAmount) FROM dbo.Archive_PosTransactions WHERE StoreID=@StoreID;
06 Phase 5 — Closure Progress Reporting
✦ I Built & Maintained This

Leadership needed weekly visibility into closure progress across all 50+ stores. I built a closure status tracking table and automated executive summary report so the program could be managed from data, not manual emails.

T-SQL — Closure Status Tracking & Executive SummarySimulated pattern
CREATE TABLE dbo.StoreClosureStatus (
  StoreID          int PRIMARY KEY,
  ClosureDate      date NULL,
  MigrationStatus  nvarchar(30) NOT NULL,   -- Pending/InProgress/Completed/Failed
  ValidationStatus nvarchar(30) NOT NULL,   -- NotStarted/Passed/Failed
  DecomApproved    bit NOT NULL DEFAULT 0,
  UpdatedAt        datetime2 NOT NULL DEFAULT SYSUTCDATETIME()
);

-- Executive snapshot query
SELECT MigrationStatus, ValidationStatus, COUNT(*) AS StoreCount
FROM dbo.StoreClosureStatus
GROUP BY MigrationStatus, ValidationStatus
ORDER BY MigrationStatus;
