Capital One's Credit Risk Management department needed to proactively monitor, analyze, and mitigate credit risk across a diverse loan portfolio. Data was scattered across 5+ heterogeneous source systems — loan application systems, customer transaction histories, credit bureau reports — with inconsistent formats and no unified analytics layer.

The team needed real-time risk visibility, automated compliance reporting, and a scalable data foundation that could handle growing volumes without degrading performance.

1
Phase 1 · 4 Weeks

Requirements & Data Understanding

Contributed — team effort

The first phase was a cross-functional discovery effort. I participated in stakeholder interviews with the Credit Risk Management team, data analysts, and compliance officers to define the key risk metrics the system needed to surface.

  • Participated in stakeholder interviews to define key credit risk metrics: loan default rates, credit utilization ratios, customer credit score distributions
  • Analyzed existing data sources across loan applications, customer transaction history, and credit bureau reports — documenting formats, volumes, and quality issues
  • Contributed to target database schema design — identifying fact tables (loan performance, risk events) and dimension tables (customer, geography, loan type, time)
  • Documented data quality issues and proposed cleansing rules for the ETL layer
Key Output From This Phase

Agreed schema design with star schema fact/dimension structure covering loan performance, customer demographics, geography, and time dimensions. This became the blueprint for my ETL and SSAS work in Phase 2 and 3.

2
Phase 2 · 6 Weeks

ETL Development & Data Warehousing

I Led This Phase

This was my primary ownership area. I engineered the SSIS ETL pipeline architecture from the ground up — designing a reusable template framework that standardized how all source systems fed into the data warehouse.

  • Designed and developed SSIS packages to automate extraction, transformation, and loading from loan application systems, customer transaction histories, and credit bureau reports
  • Built a reusable SSIS template framework with standardized error handling, logging, and retry logic — reducing future ETL development time by 40%
  • Implemented robust data validation and cleansing at the transformation stage: null checks, referential integrity enforcement, business rule validation
  • Optimized ETL pipelines for parallel processing — breaking large extracts into partitioned loads to handle increasing data volumes without extending the processing window
  • Contributed to the 10M+ record database migration — schema conversion, data type reconciliation, and validation — achieving 30% query performance improvement post-migration
  • Automated ETL workflows to guarantee consistent, timely updates to credit risk metrics — increasing operational efficiency by 50%
Impact

The reusable template framework meant that adding a new source system — what previously took a developer days — now took hours. ETL development time reduced 40% across the team for all subsequent pipeline work.

Sample 1 — ETL Staging Load with Validation & Error Logging

This pattern was central to my SSIS template framework — a stored procedure called by SSIS that handles incremental load, deduplication, business rule validation, and structured error logging in one transaction.

T-SQL · Stored Procedure Simulated · Not proprietary 
-- ═══════════════════════════════════════════════════
-- Stored Proc: usp_LoadLoanPerformanceStaging
-- Purpose:     Incremental load from source → staging
--              with validation, dedup, and error logging
-- Called by:   SSIS Execute SQL Task (Phase 2 pipeline)
-- ═══════════════════════════════════════════════════

CREATE PROCEDURE usp_LoadLoanPerformanceStaging
    @BatchDate      DATE,
    @SourceSystem   VARCHAR(50),
    @RecordsLoaded  INT OUTPUT,
    @ErrorCount     INT OUTPUT
AS
BEGIN
    SET NOCOUNT ON;
    SET XACT_ABORT ON;

    DECLARE @ProcStart  DATETIME2 = SYSDATETIME();
    DECLARE @StepName   VARCHAR(100);

    BEGIN TRY
        -- ── Step 1: Truncate today's staging partition ──
        SET @StepName = 'TruncateStaging';
        DELETE FROM stg.LoanPerformance
        WHERE BatchDate = @BatchDate
          AND SourceSystem = @SourceSystem;

        -- ── Step 2: Insert validated records ──
        SET @StepName = 'InsertValidated';
        INSERT INTO stg.LoanPerformance
               (LoanID, CustomerID, LoanType, OriginalBalance,
                CurrentBalance, DaysDelinquent, RiskScore,
                BatchDate, SourceSystem, LoadedAt)
        SELECT
               src.LoanID,
               src.CustomerID,
               src.LoanType,
               src.OriginalBalance,
               src.CurrentBalance,
               src.DaysDelinquent,
               -- Business rule: clamp risk score to valid range
               CASE WHEN src.RiskScore BETWEEN 0 AND 1000
                    THEN src.RiskScore
                    ELSE NULL
               END                                AS RiskScore,
               @BatchDate,
               @SourceSystem,
               SYSDATETIME()
        FROM   raw.LoanPerformance_Source src
        WHERE  src.BatchDate  = @BatchDate
          AND  src.LoanID    IS NOT NULL   -- null key = reject
          AND  src.CustomerID IS NOT NULL
          AND NOT EXISTS (                  -- dedup check
               SELECT 1 FROM stg.LoanPerformance ex
               WHERE ex.LoanID = src.LoanID
                 AND ex.BatchDate = @BatchDate);

        SET @RecordsLoaded = @@ROWCOUNT;

        -- ── Step 3: Count validation failures ──
        SELECT @ErrorCount = COUNT(*)
        FROM   raw.LoanPerformance_Source
        WHERE  BatchDate = @BatchDate
          AND  (LoanID IS NULL OR CustomerID IS NULL);

        -- ── Step 4: Log batch run ──
        INSERT INTO audit.ETL_BatchLog
               (ProcName, SourceSystem, BatchDate,
                RecordsLoaded, ErrorCount, DurationMs, RunAt)
        VALUES ('usp_LoadLoanPerformanceStaging', @SourceSystem,
                @BatchDate, @RecordsLoaded, @ErrorCount,
                DATEDIFF(MILLISECOND, @ProcStart, SYSDATETIME()),
                SYSDATETIME());

    END TRY
    BEGIN CATCH
        INSERT INTO audit.ETL_ErrorLog
               (ProcName, StepName, ErrorMsg, BatchDate, LoggedAt)
        VALUES ('usp_LoadLoanPerformanceStaging', @StepName,
                ERROR_MESSAGE(), @BatchDate, SYSDATETIME());
        THROW;
    END CATCH
END;

Sample 2 — Credit Risk Report Query (Before & After Optimization)

One of my key contributions was optimizing slow-running report queries. Here is the pattern I applied — rewriting correlated subqueries to window functions and adding covering indexes — achieving the 30% report generation improvement.

T-SQL · Query Optimization Simulated · Not proprietary 
/* ─────────────────────────────────────────────
   BEFORE: Correlated subquery — ran in ~8 seconds
   on 10M row table. Caused report timeout issues.
───────────────────────────────────────────── */
SELECT
    l.LoanID,
    l.CustomerID,
    l.RiskScore,
    (SELECT AVG(l2.RiskScore)         -- correlated = re-runs per row
     FROM   dw.FactLoanPerformance l2
     WHERE  l2.LoanType = l.LoanType) AS AvgRiskByType
FROM   dw.FactLoanPerformance l;

/* ─────────────────────────────────────────────
   AFTER: Window function — runs in ~1.5 seconds.
   Same result, 5x faster. Added covering index
   on (LoanType, RiskScore) to support it.
───────────────────────────────────────────── */
SELECT
    l.LoanID,
    l.CustomerID,
    l.LoanType,
    l.RiskScore,
    AVG(l.RiskScore) OVER (
        PARTITION BY l.LoanType
    )                                   AS AvgRiskByType,
    l.RiskScore - AVG(l.RiskScore) OVER (
        PARTITION BY l.LoanType
    )                                   AS RiskDeviation,
    CASE
        WHEN l.RiskScore > AVG(l.RiskScore) OVER (
             PARTITION BY l.LoanType) * 1.25
        THEN 'HIGH'
        WHEN l.RiskScore < AVG(l.RiskScore) OVER (
             PARTITION BY l.LoanType) * 0.75
        THEN 'LOW'
        ELSE 'NORMAL'
    END                                 AS RiskCategory
FROM   dw.FactLoanPerformance l
WHERE  l.BatchDate >= DATEADD(MONTH, -3, GETDATE());

Sample 3 — Power BI DAX: Default Rate & Rolling 3-Month Average

DAX · Power BI Measures Simulated · Not proprietary 
-- Default Rate % (used on main risk dashboard)
Default Rate % =
DIVIDE(
    COUNTROWS(
        FILTER(FactLoanPerformance,
               FactLoanPerformance[DaysDelinquent] >= 90)
    ),
    COUNTROWS(FactLoanPerformance),
    0
)

-- 3-Month Rolling Average Default Rate (time intelligence)
Default Rate 3M Avg =
CALCULATE(
    [Default Rate %],
    DATESINPERIOD(
        DimDate[Date],
        LASTDATE(DimDate[Date]),
        -3,
        MONTH
    )
)

-- High Risk Loan Flag (used in slicer/alert tile)
High Risk Flag =
IF(
    SELECTEDVALUE(FactLoanPerformance[RiskScore]) >
    CALCULATE(
        AVERAGE(FactLoanPerformance[RiskScore]),
        ALL(FactLoanPerformance)
    ) * 1.3,
    "⚠ HIGH RISK",
    "NORMAL"
)
