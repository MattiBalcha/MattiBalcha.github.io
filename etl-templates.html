Credit risk reporting at Capital One demands timely, consistent, and auditable data movement. Legacy one-off SSIS packages — each with different patterns, no shared logging, no reusable components — created higher failure rates, slow onboarding of new sources, and scaling problems as volume grew. This initiative replaced ad-hoc development with a production-grade reusable framework that cut both reporting latency and pipeline development time by 40%.

My direct work centered on designing the SSIS template architecture, implementing framework-grade logging and error handling, optimizing individual pipeline performance through batching and parallelism, and producing the documentation that made it adoptable across the broader team.

Framework Design Principle

Every SSIS package inherits the same control flow: Initialize → Extract → Transform/Validate → Load → Finalize. This consistency means any developer can open any package and maintain it without re-learning patterns.

02 Phase 1 — Baseline & Bottleneck Map
✦ I Led This Phase

Before redesigning anything, I established measurable baselines — end-to-end latency from source availability through staging completion to downstream report refresh. Without baselines, no performance claim is defensible.

  • Slow extracts — unindexed source queries, no predicate pushdown
  • Inefficient transforms — row-by-row operations, unnecessary sorts in SSIS data flows
  • Single-threaded loads — parallelism available but unused across independent entities
  • No incremental strategy — full reloads of large tables every run regardless of delta size
  • Silent failures — inconsistent logging meant issues took hours to triage
Output

A prioritized remediation backlog — top pipelines ranked by business impact. Targeting the highest-impact problems first is what makes 40% improvements achievable vs. random tuning.

03 Phase 2 — SSIS Golden Template Design
✦ I Directly Built This

The core deliverable: a standardized SSIS package template that all pipelines in the framework inherit. This drove the 40% dev time reduction — new sources follow the same blueprint, no reinvention needed. Parameterization meant Environment, Connection Strings, LoadMode, and BatchID were all project parameters managed via SSISDB environments — zero hardcoding inside packages.

Standard Control Flow structure:

  • Initialize_Batch — create BatchID, set parameters, write log start entry
  • PreChecks — validate source connectivity, file presence, schema expectations
  • Extract_Load_Stage — data flow tasks per entity, standardized error outputs wired
  • PostLoad_Validation — row counts, null checks, referential integrity checks
  • Finalize_Batch — write log end entry, status, summary metrics
  • OnError_Handler — centralized error capture and alerting hook
04 Phase 3 — Framework Logging & Error Handling
✦ I Directly Built This

I designed a purpose-built logging schema that captures exactly what operations need: batch status, row counts, reject counts, and structured error messages — not the verbose noise of SSIS default logging.

T-SQL — Logging SchemaSimulated pattern — no proprietary code
CREATE TABLE dbo.ETL_BatchLog (
  BatchID        bigint IDENTITY(1,1) PRIMARY KEY,
  PackageName    nvarchar(200),
  Environment    nvarchar(50),
  StartTime      datetime2,
  EndTime        datetime2,
  Status         nvarchar(20),   -- Started / Succeeded / Failed
  RowsInserted   bigint NULL,
  RowsRejected   bigint NULL,
  ErrorCount     int NULL
);
CREATE TABLE dbo.ETL_ErrorLog (
  ErrorID        bigint IDENTITY(1,1) PRIMARY KEY,
  BatchID        bigint,
  PackageName    nvarchar(200),
  TaskName       nvarchar(200),
  ErrorTime      datetime2 DEFAULT SYSUTCDATETIME(),
  ErrorMessage   nvarchar(max) NULL
);
CREATE TABLE dbo.ETL_Rejects (
  RejectID       bigint IDENTITY(1,1) PRIMARY KEY,
  BatchID        bigint,
  Entity         nvarchar(50),
  NaturalKey     nvarchar(200),
  RejectReason   nvarchar(200),
  RejectedAt     datetime2 DEFAULT SYSUTCDATETIME()
);

SSIS OnError event handlers write to ETL_ErrorLog with task name and error code. Bad data rows route via conditional split to ETL_Rejects with reason codes — every issue becomes measurable and fixable, not silent.

05 Phase 4 — Incremental Load Strategy
✦ I Directly Implemented This

Full reloads of multi-million-row staging tables were the primary latency driver. I replaced them with watermark-based incremental patterns — each feed only moves changed or new records since the last successful run.

T-SQL — Watermark Table & MERGE UpsertSimulated pattern — no proprietary code
-- Watermark table tracks last successful load per feed
CREATE TABLE dbo.ETL_Watermark (
  FeedName      nvarchar(100) PRIMARY KEY,
  LastWatermark datetime2 NOT NULL
);
-- Incremental extract using watermark
SELECT * FROM src.SourceTable
WHERE UpdatedAt > (SELECT LastWatermark
                   FROM dbo.ETL_Watermark WHERE FeedName = 'SourceFeed');
-- MERGE: update changed rows, insert new ones
MERGE dbo.Target t
USING dbo.Stage s ON t.BusinessKey = s.BusinessKey
WHEN MATCHED AND t.HashValue != s.HashValue THEN
  UPDATE SET t.Col1=s.Col1, t.Col2=s.Col2,
             t.HashValue=s.HashValue, t.UpdatedAt=SYSUTCDATETIME()
WHEN NOT MATCHED THEN
  INSERT (BusinessKey,Col1,Col2,HashValue,CreatedAt)
  VALUES (s.BusinessKey,s.Col1,s.Col2,s.HashValue,SYSUTCDATETIME());
Impact

Switching from full reloads to incremental loads was the single largest contributor to the 40% latency reduction — moving 5–10% of records per run instead of 100%.

06 Phase 5 — Parallel Processing & Modular Framework
✦ I Directly Implemented This

Independent entity loads were restructured to run concurrently using SSIS MaxConcurrentExecutables and precedence constraints — dimension loads in parallel, followed by fact loads. I also designed the modular plug-in framework: new sources onboard by inheriting the template, not writing from scratch.

T-SQL — Batch Start / End Procedures (called by SSIS Execute SQL Tasks)Simulated pattern
CREATE OR ALTER PROCEDURE dbo.usp_ETL_BatchStart
  @PackageName nvarchar(200), @Environment nvarchar(50), @BatchID bigint OUTPUT
AS
BEGIN
  INSERT dbo.ETL_BatchLog(PackageName,Environment,StartTime,Status)
  VALUES(@PackageName,@Environment,SYSUTCDATETIME(),'Started');
  SET @BatchID=SCOPE_IDENTITY();
END;
CREATE OR ALTER PROCEDURE dbo.usp_ETL_BatchEnd
  @BatchID bigint, @Status nvarchar(20),
  @RowsInserted bigint, @RowsRejected bigint
AS
BEGIN
  UPDATE dbo.ETL_BatchLog
  SET EndTime=SYSUTCDATETIME(), Status=@Status,
      RowsInserted=@RowsInserted, RowsRejected=@RowsRejected
  WHERE BatchID=@BatchID;
END;

T-SQL — Staging Validation ProcedureSimulated pattern
CREATE OR ALTER PROCEDURE dbo.usp_ETL_ValidateStaging
  @BatchID bigint, @ErrorCount int OUTPUT
AS
BEGIN
  SET NOCOUNT ON;
  DECLARE @E int = 0;
  -- Rule 1: Required fields not null
  INSERT INTO dbo.ETL_Rejects(BatchID,Entity,NaturalKey,RejectReason)
  SELECT @BatchID,'LoanFact',CAST(LoanID AS nvarchar),'Null required field'
  FROM stg.LoanFact
  WHERE BatchID=@BatchID
    AND (LoanID IS NULL OR RiskScore IS NULL OR AsOfDate IS NULL);
  SET @E+=@@ROWCOUNT;
  -- Rule 2: Risk score within valid range (300–850)
  INSERT INTO dbo.ETL_Rejects(BatchID,Entity,NaturalKey,RejectReason)
  SELECT @BatchID,'LoanFact',CAST(LoanID AS nvarchar),'RiskScore out of range'
  FROM stg.LoanFact
  WHERE BatchID=@BatchID AND (RiskScore<300 OR RiskScore>850);
  SET @E+=@@ROWCOUNT;
  SET @ErrorCount=@E;
END;
