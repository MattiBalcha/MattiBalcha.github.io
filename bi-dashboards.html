Credit risk teams need trusted, consistent KPIs — default rates, exposure by segment, risk score distribution — and the ability to drill from portfolio level down to individual loan detail without manual data pulls. This project delivered a complete BI stack: SSAS semantic model, Power BI dashboards, and SSRS paginated compliance outputs, all backed by validated measures that reconciled to the same SQL source of truth.

My contributions spanned semantic modeling, DAX measure development, Power Query standardization, and reconciliation validation. I worked closely with the Power BI team and stakeholders to ensure the dashboards answered real operational questions, not just displayed data.

Design Context

The SSAS semantic model was the single source of truth for all KPI definitions. Both Power BI (interactive) and SSRS (paginated) connected to the same model, ensuring consistency across all consumers.

02 Phase 1 — KPI Definition & Semantic Layer Design
◈ Contributed — Collaborated with BI Team

Before building any visuals, I worked with stakeholders to lock down KPI definitions — the exact numerator/denominator logic for default rate, the risk band thresholds, the time grain for trend analysis. Undefined or inconsistent metrics cause "dashboard wars" where different teams report different numbers from the same data.

  • Default Rate (%) — confirmed as (loans with 90+ day delinquency) / (total active loans) at month-end grain
  • Risk Exposure — sum of outstanding balances in High/Very High risk bands
  • Score Distribution — segmented into 4 standard credit score bands (Poor/Fair/Good/Very Good)
  • Anomaly Indicator — segment whose default rate exceeds portfolio average by >30%
03 Phase 2 — SSAS Cube / Tabular Model
✦ I Directly Worked On This

Built SSAS dimensions and hierarchies that support multi-level risk analysis. Geography hierarchy (Region → State), Demographics (Age Band / Income Band / Score Band), Loan Type hierarchy, and Time (Date → Month → Quarter → Year). Calculated measures and KPIs sat in the semantic layer — one source of truth that feeds both Power BI and SSRS.

DAX Measures — Power BI / Tabular ModelSimulated pattern — no proprietary code
-- Total and defaulted loan counts
Total Loans :=
DISTINCTCOUNT( Loans[Loan_ID] )

Defaulted Loans :=
CALCULATE(
    DISTINCTCOUNT( Loans[Loan_ID] ),
    Loans[Default_Flag] = 1
)

-- Default Rate %
Default Rate % :=
DIVIDE( [Defaulted Loans], [Total Loans] )

-- Risk exposure in high-risk bands
Risk Exposure :=
CALCULATE(
    SUM( Loans[Outstanding_Balance] ),
    Loans[Risk_Band] IN { "High", "Very High" }
)

-- 3-Month rolling average default rate
Rolling 3M Default Rate :=
CALCULATE(
    [Default Rate %],
    DATESINPERIOD( DimDate[Date], LASTDATE(DimDate[Date]), -3, MONTH )
)

-- High-risk segment flag (30% above portfolio average)
High Risk Segment :=
IF( [Default Rate %] > [Default Rate %] * 1.3, "High Risk", "Normal" )
04 Phase 3 — Power BI Dashboard Build
◈ Contributed — Collaborated with Power BI Team

Dashboard pages were organized by analytical persona. Executive overview for KPI monitoring, analyst investigation for drill-down, and compliance summary for scheduled regulatory outputs. I contributed to data model design, measure validation, and refresh configuration.

Power Query M — Score Banding & StandardizationSimulated pattern
let
  Source = Sql.Database("SERVER", "CreditRiskDB"),
  Loans = Source{[Schema="dbo",Item="Loans"]}[Data],
  Cleaned = Table.TransformColumns(Loans, {{"Region", Text.Trim, type text}}),
  ScoreBand = Table.AddColumn(Cleaned, "Score_Band", each
      if [Credit_Score] < 580 then "Poor"
      else if [Credit_Score] < 670 then "Fair"
      else if [Credit_Score] < 740 then "Good"
      else "Very Good"
  , type text)
in
  ScoreBand
Why 25% Faster Anomaly Detection

Consistent cube-backed measures + drill-through from portfolio to segment to loan level reduced the time between "I see something wrong" and "I know which segment and why" from hours to minutes. That operational speed improvement is what the 25% metric reflects.

05 Phase 4 — Validation & Reconciliation
✦ I Directly Worked On This

Every key measure was reconciled back to SQL base tables before any dashboard was marked production-ready. This is what "20% improved BI accuracy" means — not a guess, but a closed loop between the semantic model and the source-of-truth data.

T-SQL — Dashboard Reconciliation PatternSimulated pattern
-- Reconcile default rate between dashboard and base tables
SELECT
  CAST(SUM(CASE WHEN Default_Flag = 1 THEN 1 ELSE 0 END) AS float) /
  NULLIF(COUNT(*), 0)                 AS DefaultRate,
  SUM(Outstanding_Balance)             AS TotalExposure,
  AVG(CAST(Credit_Score AS float))  AS AvgScore
FROM dbo.Loans
WHERE ReportMonth = @ReportMonth;
